1. Visión General del Sistema

El sistema opera en dos etapas independientes: primero se enrolla al objetivo (construir una galería de features), y luego se busca ese objetivo en videos o en un stream en vivo.

Video de referencia .mp4 / .avi
→
enroll_launcher.py GUI tkinter
→
auto_enroll_v2.py SAMURAI + FeatureExtractor
→
galería.json + .npy Data/
Videos / Stream *.mp4 / RTSP
→
reid_viewer_v2_offline.py o _stream.py
→
galería.json + .npy Data/
→
Clips + consola reid_clips/

Flujo interno de enrollment (auto_enroll_v2.py)

  1. LocateAnything-3B — localiza al objetivo en el primer frame válido con un bbox inicial
  2. SAMURAI (SAM2 con memoria) — trackea al objetivo cuadro a cuadro durante search_frames frames
  3. LOCK — una vez que el tracker está seguro, graba clip_frames frames del objetivo bloqueado
  4. Extracción de features — para cada crop bloqueado: aplica SR opcional → extrae vector de características (TransReID / OSNet / ViT / CLIP)
  5. Filtrado outliers — cosine similarity vs. ancla; se eliminan crops de baja similitud (IQR fence)
  6. Top-K — selecciona los top_crops mejores crops por similitud
  7. PCA visualization — grafica la distribución 2D de features (guardado como _pca.png)
  8. Data augmentation — genera variaciones (flip, brillo, saturación, crop) según aug_factor
  9. Guardar galería — escribe Data/{nombre}.json + Data/{nombre}.npy

Flujo interno de búsqueda (reid_viewer_v2_offline.py)

  1. Fase 1 — YOLO11x batch — detecta todas las personas/vehículos en cada frame analizado; extrae features en lote (GPU)
  2. Galería negativa — los primeros neg_frames frames construyen un set de "no es el objetivo" para rechazar falsos positivos
  3. [TOP] diagnóstico — imprime el mejor score/fracción encontrado en toda la fase 1 para ajuste de umbrales
  4. Fase 2 — Máquina de estados — simula la lógica online (SEARCH → TRACK → LOST) sobre las detecciones de fase 1
  5. Fase 3 — Clips + crops — para cada intervalo TRACK genera un .mp4 recortado a la zona bloqueada

2. enroll_launcher.py — Interfaz Gráfica

GUI de enrollment — lanzador visual de auto_enroll_v2.py
Permite configurar y ejecutar el enrollment sin escribir comandos a mano
python enroll_launcher.py

Secciones de la interfaz

Objetivo

  • Video — ruta al video de referencia (Browse o pegar path). Al seleccionar un video válido aparece una previsualización del frame (~5% del video) con resolución, fps y duración.
  • Descripción — texto libre del objetivo. Ej: "Hombre en moto con playera negra y pantalón azul". Se usa para auto-generar el nombre de la galería y para que LocateAnything-3B entienda qué buscar.
  • Nombre galería — nombre del archivo JSON/NPY resultante. Auto-generado desde la descripción, editable manualmente.
  • Agregar a — si ya existe una galería, agregar los nuevos features a ella (útil para robustecer con otro ángulo).

Parámetros

ControlDefaultDescripción
Modelo featuresautoBackend de features. auto elige el mejor disponible (TransReID > OSNet). Ver sección de modelos.
Aug factor5Cuántas versiones aumentadas por crop base (flip, brillo, contraste, saturación). Mayor = galería más robusta.
Top crops5Cuántos crops base se incluyen en la galería final (los N con mejor cosine similarity).
Veh ratio1.0Escala del crop de vehículo (modo Contexto). 1.0 = sin escala extra. 1.5–2.0 si la persona es pequeña respecto al vehículo.
Frames búsqueda120SAMURAI analiza hasta N frames antes de decidir el LOCK del objetivo.
Frames clip100Una vez bloqueado, graba N frames del objetivo para construir la galería.
Video completofalseIgnorar los límites de frames; analizar todo el video.
Super-resoluciónoffAplica Real-ESRGAN antes de extraer features. Muy útil si la cámara está lejos y los crops son pequeños (<60×30 px).
Escala SR4Factor de upscale: x2 (más rápido) o x4 (mejor calidad).

Clases YOLO

Buscar — clases que YOLO detecta en el frame completo (por defecto: person (0)). Se sugieren automáticamente desde la descripción.

Contexto — clases que deben estar cerca para confirmar la identidad. Ej: para una moto, marcar motorcycle (3). El crop se expandirá 80% arriba y 30% a los lados del vehículo para capturar al rider.

Importante: Las clases se auto-detectan desde palabras clave en la descripción (moto, carro, bicicleta…). Siempre verifica que la auto-detección sea correcta antes de ejecutar.

Botones post-enrollment

  • 🖼 Ver Crops — abre una ventana con el grid scrollable de todos los crops base y aumentados. Los crops base se muestran con el bbox en crudo (sin letterbox).
  • 🔍 Ver PCA — abre el gráfico PCA 2D de los features del enrollment (verde = conservados, rojo = outliers eliminados).

Ninguna ventana se abre automáticamente al terminar — debes hacer clic en el botón que quieras.

3. auto_enroll_v2.py — Motor de Enrollment

Normalmente se invoca desde enroll_launcher.py — raramente a mano

Diagrama de bloques

python auto_enroll_v2.py \
  --video /ruta/video.mp4 \
  --description "Hombre con playera amarilla y pantalón negro" \
  --name Playera_Amarilla_Pantaon_20260604 \
  --feat_model auto \
  --aug_factor 5 --top_crops 5 \
  --search_frames 120 --clip_frames 100 \
  --sr --sr_scale 4 --sr_blend 0.5 \
  --no_show_pca

Parámetros principales

4. reid_viewer_v2_offline.py — Búsqueda en Videos Grabados

Modo batch — máxima velocidad con GPU (13–20 f/s en RTX 3060)
Procesa carpetas de videos, genera clips de cada intervalo donde aparece el objetivo

Diagrama de bloques

Uso básico

python reid_viewer_v2_offline.py \
  --videos_dir /ruta/Videos \
  --gallery Data/Playera_Amarilla_Pantaon_20260604.json \
  --thr_feat 0.70 --min_fraction 0.25 \
  --frame_skip 2 --yolo_batch 8 --feat_batch 32 \
  --output_dir reid_clips

# Motos (TransReID, modo Contexto)
python reid_viewer_v2_offline.py \
  --videos_dir /ruta/Videos \
  --gallery Data/Dos_Personas_Motocicleta_20260604.json \
  --thr_feat 0.60 --min_fraction 0.25 \
  --frame_skip 2 --yolo_batch 8 --feat_batch 32 \
  --output_dir reid_clips

# Carros (ViT/DINO)
python reid_viewer_v2_offline.py \
  --videos_dir /ruta/Videos \
  --gallery Data/Taxi_Color_Rosa_20260604.json \
  --thr_feat 0.60 --min_fraction 0.25 \
  --frame_skip 2 --yolo_batch 8 --feat_batch 32 \
  --output_dir reid_clips

Fase 1 — Qué significa cada línea

[F1]  451f → 226 analizados  2078 dets  32.5s  (14fps)   ← resumen de YOLO+FE
[NEG] 80 muestras negativas                                ← galería negativa lista

8. Máquina de Estados: SEARCH → TRACK → LOST

🔍 SEARCH
YOLO detecta en frame completo.
Para cada detección se calcula score y frac_s.
Si frac_s ≥ min_fraction y no es rechazado por neg_feats → candidato.
Se acumulan en confirm_count si el candidato permanece en zona zone_thr px.

→ TRACK cuando se cumple algún mecanismo de lock (ver abajo).
→
🎯 TRACK
YOLO detecta solo en la zona bloqueada (bounding box expandido).
Umbral más permisivo: thr_feat_track = thr_feat × 0.85.
Zona se desplaza según la detección activa (velocidad predictiva).
lost_count sube cuando no hay detección válida.

→ SEARCH si lost_count ≥ lost_max.
→
❌ LOST / SEARCH
Zona perdida tras lost_max frames sin detección válida.
El intervalo TRACK se cierra y se genera el clip de video.
Se imprime [LOST] con avg_score y avg_frac del intervalo.

→ Vuelve a SEARCH inmediatamente.

Mecanismos de lock SEARCH → TRACK

TipoCondiciónCuándo se activa
[reg] Regular confirm_count ≥ lock_frames_eff Y regular_lock_spd_ok Caso general: N frames consecutivos con buena confianza y/o movimiento detectado.
[imm] Inmediato confirm_count ≥ 2 Y frac ≥ 70% Y score ≥ thr × 1.10 Match muy alto en el frame actual. Evita que decoys de frac baja contaminen confirm_min_frac.
[dyn] Dinámico confirm_count ≥ 3 Y frac ≥ 55% Y spd ≥ 2.0 px/f Y frac_ok Objetivo en movimiento rápido con match razonable.

lock_frames_eff — frames efectivos de confirmación

Condiciónlock_frames_eff
confirm_min_frac ≥ 99% (con neg_feats)lock_frames (4)
confirm_min_frac ≥ 99% (sin neg_feats)lock_frames × 5 (20)
frac_ok: min_frac ≥ 70% ó best_frac ≥ 80%lock_frames (4)
ninguno de los anterioreslock_frames × 2 (8)
Problema frecuente — "decoy envenenando confirm_min_frac":
Si hay otra persona con ropa similar (ej. también roja), puede aparecer en un frame intermedio de la ventana de confirmación con frac=35%. Esto baja confirm_min_frac a 35%, deshabilita frac_ok y duplica lock_frames_eff a 8. El lock inmediato [imm] fue diseñado para evitar esto: si el frame actual tiene frac ≥ 70% y score ≥ thr × 1.10, lockea en 2 frames sin importar el historial.

10. Problemas Comunes y Soluciones

❌ [TOP] muestra f=0% en todos los videos

La galería fue enrollada con crops incorrectos (portrait con relleno negro en lugar de el crop en crudo). Solución: re-enrollar con la versión corregida de auto_enroll_v2.py.

Diagnóstico: python3 -c "import cv2; img=cv2.imread('Data/Frames/galeria_fXXXXXX.jpg'); print(img.shape)" — si shape es (256,128,3) = portrait con letterbox, re-enrollar.

❌ [TOP] OK ≥ thr pero NO ENCONTRADO

La fase 1 encontró detecciones buenas, pero la máquina de estados no lockea. Causas probables:

  • Decoy de frac baja: otro objeto con ropa similar aparece en frames intermedios y arrastra confirm_min_frac hacia abajo. El lock [imm] debería resolverlo si score ≥ thr × 1.10.
  • Objetivo muy estático: spd=0 y confirm_min_frac < 0.85. Bajar --lock_frames a 2 o 3.
  • zone_thr demasiado pequeño: el objetivo oscila entre frames y cae fuera de la zona de confirmación. Subir --zone_thr a 120.
  • Umbral demasiado alto: bajar --thr_feat en 0.05 incrementos.
❌ Muchos falsos positivos (lockea en personas incorrectas)
  • Subir --thr_feat y/o --min_fraction.
  • Subir --neg_frames a 120 o más para que la galería negativa capture más decoys.
  • Re-enrollar con descripción más específica o video de mejor calidad.
❌ Motocicleta no detectada / crops en portrait con relleno negro

Verificar que en el enrollment se marcó motorcycle (3) como clase de Contexto. El crop debe ser del bbox expandido de la moto (80% arriba, 30% laterales) y guardarse en crudo (sin letterbox). Si el crop sigue siendo 256×128, re-enrollar.

❌ Error: expected floating-point number but got ""

Error de tkinter al limpiar el campo de Veh Ratio o Frames en enroll_launcher. El launcher ya maneja esto con try/except. Si persiste, actualizar enroll_launcher.py.

❌ PCA: ventana bloquea el enrollment

Si se invoca auto_enroll_v2.py directamente sin --no_show_pca, la ventana de PCA abre un root.mainloop() que bloquea hasta cerrarla. El launcher siempre pasa --no_show_pca automáticamente.

Flujo de diagnóstico rápido
  1. Ejecutar con --debug para ver el score y fracción de cada detección individual.
  2. Revisar [TOP]: si FALTA, bajar --thr_feat. Si f=0%, re-enrollar.
  3. Si OK ≥ thr pero no lockea, añadir --debug y buscar [SEARCH] primer match. Ver si confirm llega a 2 y si [LOCK] aparece.
  4. Si sigue sin lockear con 2 confirmaciones, bajar --lock_frames 2 y revisar zone_thr.
